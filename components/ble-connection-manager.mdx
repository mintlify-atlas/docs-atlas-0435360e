---
title: BLE Connection Manager
description: Stack-agnostic API for managing BLE connections, advertising, scanning, GATT operations, and security on ESP32 series devices.
---

The `ble_conn_mgr` component provides a simplified, stack-agnostic interface for common BLE operations. It supports peripheral and central roles, multi-connection, extended advertising, L2CAP CoC channels, and Security Manager pairing.

**Supported SoCs:** ESP32, ESP32-C2, ESP32-C3, ESP32-S3

## Installation

```bash
idf.py add-dependency "espressif/ble_conn_mgr"
```

<Note>
Requires ESP-IDF ≥ 4.3. The component wraps either NimBLE or Bluedroid depending on your sdkconfig.
</Note>

## Configuration struct

`esp_ble_conn_config_t` is passed to `esp_ble_conn_init()` to configure the device identity and advertising payload.

| Field | Type | Description |
|---|---|---|
| `device_name` | `uint8_t[29]` | Local name broadcast when advertising (peripheral role). Max 29 bytes. |
| `remote_name` | `uint8_t[29]` | Target device name to connect to when `scan_cb` is NULL (central role). |
| `broadcast_data` | `uint8_t[15]` | Manufacturer-specific data included in advertising. |
| `extended_adv_len` | `uint16_t` | Extended advertising data length. |
| `periodic_adv_len` | `uint16_t` | Periodic advertising data length. |
| `extended_adv_rsp_len` | `uint16_t` | Extended advertising response data length. |
| `extended_adv_data` | `const char *` | Extended advertising data payload. |
| `periodic_adv_data` | `const char *` | Periodic advertising data payload. |
| `extended_adv_rsp_data` | `const char *` | Extended advertising response data payload. |
| `include_service_uuid` | `uint16_t : 1` | Set to 1 to include a service UUID in advertising. |
| `adv_uuid_type` | `esp_ble_conn_uuid_type_t` | UUID type: `BLE_CONN_UUID_TYPE_16` or `BLE_CONN_UUID_TYPE_128`. |
| `adv_uuid16` / `adv_uuid128` | union | Service UUID value (16-bit or 128-bit). |

|---|---|---|

## Events

Events are delivered via the ESP event loop on the `BLE_CONN_MGR_EVENTS` base. Register a handler with `esp_event_handler_register`.

| Event | Value | Triggered when |
|---|---|---|
| `ESP_BLE_CONN_EVENT_UNKNOWN` | 0 | Unknown / unhandled event |
| `ESP_BLE_CONN_EVENT_STARTED` | 1 | Connection manager has started |
| `ESP_BLE_CONN_EVENT_STOPPED` | 2 | Connection manager has stopped |
| `ESP_BLE_CONN_EVENT_CONNECTED` | 3 | A new BLE connection is established |
| `ESP_BLE_CONN_EVENT_DISCONNECTED` | 4 | A connection is terminated |
| `ESP_BLE_CONN_EVENT_DATA_RECEIVE` | 5 | A notification or indication is received |
| `ESP_BLE_CONN_EVENT_DISC_COMPLETE` | 6 | GATT service discovery is complete |
| `ESP_BLE_CONN_EVENT_PERIODIC_REPORT` | 7 | Periodic advertising report received |
| `ESP_BLE_CONN_EVENT_PERIODIC_SYNC_LOST` | 8 | Periodic sync lost |
| `ESP_BLE_CONN_EVENT_PERIODIC_SYNC` | 9 | Periodic sync established |
| `ESP_BLE_CONN_EVENT_CCCD_UPDATE` | 10 | CCCD written by peer (notify/indicate toggled) |
| `ESP_BLE_CONN_EVENT_MTU` | 11 | GATT MTU negotiated or updated |
| `ESP_BLE_CONN_EVENT_CONN_PARAM_UPDATE` | 12 | Connection parameters updated |
| `ESP_BLE_CONN_EVENT_SCAN_RESULT` | 13 | Advertisement received during central scan |
| `ESP_BLE_CONN_EVENT_ENC_CHANGE` | 14 | Link encryption state changed |
| `ESP_BLE_CONN_EVENT_PASSKEY_ACTION` | 15 | Passkey input, display, or comparison required |

The event data is delivered as `esp_ble_conn_event_data_t *` in the `event_data` argument of the handler. Access the correct union member based on the event type (e.g., `data->connected.conn_handle`, `data->disconnected.reason`, `data->data_receive`, `data->passkey_action.action`).

## Code example

<Steps>
  <Step title="Initialize and start">

```c
#include "esp_ble_conn_mgr.h"
#include "esp_event.h"
#include "esp_log.h"

static const char *TAG = "ble_example";

static void ble_event_handler(void *arg, esp_event_base_t base,
                              int32_t event_id, void *event_data)
{
    esp_ble_conn_event_data_t *data = (esp_ble_conn_event_data_t *)event_data;

    switch ((esp_ble_conn_event_t)event_id) {
    case ESP_BLE_CONN_EVENT_CONNECTED:
        ESP_LOGI(TAG, "Connected, handle=0x%04x", data->connected.conn_handle);
        break;
    case ESP_BLE_CONN_EVENT_DISCONNECTED:
        ESP_LOGI(TAG, "Disconnected, reason=0x%04x", data->disconnected.reason);
        break;
    case ESP_BLE_CONN_EVENT_DATA_RECEIVE:
        ESP_LOGI(TAG, "Data received, len=%d", data->data_receive.data_len);
        break;
    default:
        break;
    }
}

void app_main(void)
{
    // Required before using the event loop
    ESP_ERROR_CHECK(esp_event_loop_create_default());

    // Register event handler
    ESP_ERROR_CHECK(esp_event_handler_register(BLE_CONN_MGR_EVENTS,
                                               ESP_EVENT_ANY_ID,
                                               ble_event_handler, NULL));

    // Configure and initialize
    esp_ble_conn_config_t config = {
        .device_name = "ESP_BLE_DEVICE",
        .broadcast_data = CONFIG_EXAMPLE_BLE_SUB_ADV,
    };
    ESP_ERROR_CHECK(esp_ble_conn_init(&config));

    // Start advertising (peripheral) or scanning (central)
    ESP_ERROR_CHECK(esp_ble_conn_start());
}
```

  </Step>
  <Step title="Send a notification">

```c
// Build payload and notify all connected centrals
uint8_t payload[] = {0x01, 0x02, 0x03};

esp_ble_conn_data_t notify_data = {
    .type  = BLE_CONN_UUID_TYPE_16,
    .uuid  = { .uuid16 = 0xFF01 },
    .data  = payload,
    .data_len = sizeof(payload),
};
ESP_ERROR_CHECK(esp_ble_conn_notify(&notify_data));
```

  </Step>
  <Step title="Tear down">

```c
ESP_ERROR_CHECK(esp_ble_conn_stop());
ESP_ERROR_CHECK(esp_ble_conn_deinit());
```

  </Step>
</Steps>
